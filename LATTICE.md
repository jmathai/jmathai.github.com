# LATTICE.md

Repo guide for the Lattice editor's backend AI agent. Read this on every change request.
The request payload is `{page_url, selector, element_html, instruction}`; the goal is a
correct source edit and commit in **one shot**, no exploratory grepping beyond what this
file anticipates.

---

## 1. Purpose

`lattice.js` runs on the published site, lets the visitor pick a DOM element, and POSTs
`{repo, annotation: {page_url, selector, element_html, instruction}}` to the backend. The
backend invokes you, the coding agent. You must:

1. Resolve `page_url` to the source file(s) that produce that page.
2. Resolve `selector` + `element_html` to the specific source region (content file,
   layout, include, or stylesheet).
3. Apply the change described by `instruction`, preserving Liquid templating, front
   matter, and surrounding markup.
4. Verify the change lands by running the Jekyll build and confirming the modified file
   is the one in the load path for that URL.

`element_html` is your strongest clue — a distinctive substring grepped across source
roots almost always lands on the correct file. Use it first.

---

## 2. Stack

- **Static site generator:** Jekyll (Ruby), pinned via `Gemfile` / `Gemfile.lock`.
- **Markdown:** kramdown (`_config.yml` → `markdown: kramdown`). Posts/pages mix
  markdown, raw HTML, and Liquid in the same file.
- **Templating:** Liquid (`{{ ... }}` and `{% ... %}`).
- **Permalink scheme:** `permalink: /:categories/:title/` in `_config.yml` for posts;
  pages set `permalink:` explicitly in front matter.
- **Styling:** site-wide CSS lives at `assets/less/main.css` (loaded by
  `_includes/head.html`). The `Gruntfile.js` Less→CSS pipeline exists but is dormant —
  see §3 and §7.3.
- **Theme origin:** `so-simple-theme` by mmistakes (`package.json`), customized.

## 3. Source roots vs build artifacts

**Edit these (source):**

| Path | What it is |
| --- | --- |
| `_posts/` | Blog posts. Filename is `YYYY-MM-DD-slug.md`; permalink derives from front matter + `_config.yml`. |
| `_layouts/` | Page chrome. Three layouts: `page.html`, `post.html`, `job.html`. |
| `_includes/` | Partials: `head.html`, `navigation.html`, `footer.html`, `scripts.html`, `browser-upgrade.html`, `disqus_comments.html`, `startup-blurb.html`, plus `_includes/job/work-at-*.html` (job page bodies). |
| `*.md`, `*.html` at the repo root | Top-level pages (`about.md`, `articles.html`, `generosity.html`, `openphoto-trovebox.md`, `download-elodie.md`, `theme-setup.md`, `404.md`, `index.html`). |
| `job/*.md` | Job-page stubs. The body lives in `_includes/job/work-at-<name>.html`, not in the `.md`. |
| `assets/less/main.css` | Site-wide stylesheet actually loaded by `_includes/head.html`. Edit this for design changes on post/page layouts. |
| `assets/home/css/` | Stylesheets for the homepage (`index.html`) only. |
| `assets/job/bootstrap.min.css` + inline `<style>` in `_layouts/job.html` | Style sources for job pages. |
| `assets/js/_*.js`, `assets/js/plugins/*.js`, `assets/js/vendor/*.js` | JS sources. The minified bundle is referenced below — see §7.3. |
| `_config.yml` | Site config (title, owner, social, nav `links:`, permalink scheme). |
| `images/` | All image assets referenced from content/layouts. |

**Never edit these (generated or out of load path):**

| Path | Why |
| --- | --- |
| `_site/` | Jekyll build output. Overwritten on every build. |
| `.jekyll-cache/` | Jekyll incremental cache. |
| `assets/less/*.less` | Less sources for the original theme. **Not compiled in this repo.** The loaded CSS is `assets/less/main.css`. |
| `assets/css/main.css` | Stale theme copy; **not** in the load path. Do not edit. |
| `assets/js/scripts.min.js` | Generated by Grunt uglify; the Grunt pipeline is dormant here. Edit `assets/js/_main.js` and rebuild only if you've confirmed the agent operator wants the bundle regenerated; otherwise treat this as a binary you don't touch. |
| `node_modules/`, `vendor/` | Tooling. |
| `lattice.js` (symlink at repo root) | Editor script bundle; ships from an external worktree. |

If you're tempted to edit a file in this column, stop and re-resolve from the URL.

## 4. URL → source-file resolution

Apply rules in order; first match wins.

1. **Job page** — URL matches `/work-at-<name>`:
   - Stub: `job/<name>.md` (front matter only; sets `layout: job`, `permalink`,
     `title`).
   - Body content: `_includes/job/work-at-<name>.html` (this is where you edit text /
     structure of the job page).
   - Routing: `_layouts/job.html` uses a `{% case page.url %}` block to include the
     correct partial. **When adding a new job page, you must also add a `{% when
     '/work-at-<name>' %}` arm.**

2. **Post** — URL matches the permalink pattern `/:categories/:title/`:
   - Source is `_posts/YYYY-MM-DD-<title-slug>.md`. The slug is the file's
     `title`-derived URL segment; the category segment in the URL comes from the front
     matter `category:` (e.g. `category: generosity` → `/generosity/<slug>/`).
   - If there's no category in front matter, the URL is `/<slug>/`.
   - Layout: `_layouts/post.html`.

3. **Top-level page** — URL is something like `/about/`, `/articles/`, `/generosity/`,
   `/openphoto-trovebox/`, `/download-elodie/`, `/theme-setup/`:
   - Source is a root-level `.md` or `.html` with matching `permalink:` in front
     matter. `articles.html` and `generosity.html` are index pages that loop over
     `site.posts` / `site.categories.generosity`.
   - Layout: usually `_layouts/page.html`.

4. **Homepage** — URL is `/`:
   - Source is `index.html` at the repo root. It is a **standalone** HTML document — it
     does **not** use a Jekyll layout and has its own `<head>` and stylesheets in
     `assets/home/css/`. Edits to the homepage go directly in `index.html` and/or
     `assets/home/css/style.css`.

If the URL doesn't match any of the above, `grep -r "permalink: <url>" --include='*.md'
--include='*.html'` across the repo (excluding `_site/`). The matching file is the source.

## 5. Page anatomy (post / page / job layouts)

A rendered post/page is composed top-down as:

```
<html>
  <head>
    {% include head.html %}          ← _includes/head.html (meta, css, fonts, favicon)
    [inline <style> if post layout]   ← _layouts/post.html lines 8–26 only
  </head>
  <body>
    {% include navigation.html %}     ← _includes/navigation.html (masthead + nav)
    <div id="main">
      <article>
        <header class="entry-header">  ← title, post-nav links (layout)
        <footer class="entry-meta">    ← author, date, share (post layout only)
        <div class="entry-content">
          {{ content }}                ← page/post body from the content file
        </div>
      </article>
    </div>
    <footer>
      {% include footer.html %}       ← _includes/footer.html (copyright, social icons)
    </footer>
    {% include scripts.html %}        ← _includes/scripts.html (jquery, bundle, lattice.js)
  </body>
</html>
```

The job layout (`_layouts/job.html`) is different: it has its own `<head>`/`<style>`,
loads `assets/job/bootstrap.min.css`, and the body is a `{% case page.url %}` that
includes `_includes/job/work-at-<name>.html` directly. There is no `_includes/head.html`,
`navigation.html`, `footer.html`, or `scripts.html` on job pages. `lattice.js` is wired
in inline at the bottom of the layout.

The homepage (`index.html`) is its own self-contained document — none of the above
applies.

## 6. Selector → source-region heuristic

Run this decision procedure with `selector`, `element_html`, and the layout family from §4.

1. **Grep `element_html` first.** Pick the most distinctive substring (a phrase, an
   `href`, an `alt`, a class name with adjacent text). Search source roots in this
   order:
   - The content file the URL resolved to (§4).
   - `_includes/job/` if it's a job page.
   - `_layouts/` matching the URL's layout.
   - `_includes/` in general.
   - `assets/less/main.css` (or `assets/home/css/style.css` for homepage,
     `_layouts/job.html`'s `<style>` for jobs) — for class definitions.

   If one source file contains the substring, you have your edit target. Stop searching.

2. **If `element_html` is too generic to grep** (e.g., a `<div>` with only Liquid output
   inside), classify the selector by region:

   | Selector shape / location on page | Source region |
   | --- | --- |
   | `head meta`, `head title`, `head link`, favicon | `_includes/head.html` (post/page) or the `<head>` of `_layouts/job.html` / `index.html` |
   | `.masthead`, `.site-logo`, `.site-title`, `.site-description`, `#site-nav`, `.navigation-wrapper` | `_includes/navigation.html`. Note the `<ul>` nav links are currently commented out; uncomment if you need to surface them. The `links:` array in `_config.yml` is what they'd iterate over. |
   | `article.entry > header.entry-header`, `.entry-title`, post share/meta/`.post-nav-links` | `_layouts/post.html` (for posts) or `_layouts/page.html` (for pages). |
   | Anything inside `.entry-content` | The content file the URL resolved to (the post's `.md`, the page's `.md`/`.html`). |
   | `.footer-wrapper`, `footer[role=contentinfo]`, `.social-icons` | `_includes/footer.html`. Social-icon visibility is driven by `_config.yml`'s `owner:` block. |
   | `<script>` tags, search UI | `_includes/scripts.html`. |
   | Job page `<header>`, `<section>`, `<h2>`, paragraphs | `_includes/job/work-at-<name>.html`. |
   | Job page layout shell (background, container, fonts, page title rendering) | `_layouts/job.html` (`<head><style>` block or the `<body>` wrapper). |
   | Homepage anything (`/`) | `index.html` and/or `assets/home/css/style.css`. |
   | A CSS class definition (color, spacing, typography on a post/page) | `assets/less/main.css`. Search for the class literal there. |

3. **If still ambiguous**, prefer editing the most specific source: a single content
   file beats a layout; a layout beats an include shared by multiple pages. Edit the
   include only when the change is intended to apply site-wide.

## 7. Change-type playbooks

### 7.1 Text edit

1. Grep the visible text from `element_html` (drop Liquid output, trim whitespace).
2. Account for markdown rendering — the source may have `**bold**` for `<strong>`,
   `*em*` for `<em>`, `[text](url)` for `<a>`, `#`/`##` for headings, blank-line
   paragraph breaks (no `<p>` in source).
3. Replace the text in place. Preserve surrounding Liquid (`{{ ... }}`,
   `{% ... %}`) and front matter exactly.
4. If the match is in `_config.yml` (e.g., site title, owner name, nav link label), edit
   there — those values are referenced from layouts/includes.

### 7.2 Structural change (move, add, remove)

1. Decide the scope: page-local content (`{{ content }}` body) vs chrome
   (layout/include).
2. Page-local: edit the content file. Markdown ordering is the source-of-truth ordering.
3. Chrome: edit the layout or include. Preserve all Liquid control flow — wrap new
   elements outside `{% if %}` / `{% for %}` blocks unless you intend them to be
   conditional.
4. New job page: add `job/<name>.md` (stub: `layout: job`, `permalink: /work-at-<name>`,
   `title:`), create `_includes/job/work-at-<name>.html` (body), and add a
   `{% when '/work-at-<name>' %}` arm to the `{% case page.url %}` in
   `_layouts/job.html`. Missing the case arm means the include is never rendered.
5. Removing an element: also remove any Liquid that references it (e.g., a footer social
   icon block keyed off `site.owner.X` is gated by `{% if site.owner.X %}` — clear the
   value in `_config.yml` instead of deleting the markup if the goal is to hide it).

### 7.3 Design / style edit

Identify which family the URL belongs to (§4), then edit the right stylesheet:

- **Post or page layout** (`/articles/`, `/about/`, post permalinks, etc.):
  edit **`assets/less/main.css`** directly. This is the file `_includes/head.html`
  loads. Do **not** edit the `.less` files in the same directory — they are
  uncompiled theme sources, not in the load path.
- **Job pages** (`/work-at-*`):
  - Layout-shell rules (typography, container, headings) live in the inline `<style>`
    block inside `_layouts/job.html` (lines ~15–51). Edit there for changes that should
    apply to all job pages.
  - Bootstrap reset comes from `assets/job/bootstrap.min.css` — treat as vendor; don't
    edit.
  - Per-page style: add a scoped `<style>` block at the top of
    `_includes/job/work-at-<name>.html` only if the change should apply to a single job
    page.
- **Homepage** (`/`):
  edit `assets/home/css/style.css` (or `reset.css` for resets). Do not edit
  `assets/less/main.css` for homepage changes — the homepage doesn't load it.

When in doubt, find the class in `element_html`, grep the CSS files referenced by the
page's `<head>` (or `_includes/head.html` for Jekyll-rendered pages), and edit the file
where the rule is currently defined. Add a new rule at the bottom of the same file when
the selector isn't yet styled.

## 8. Liquid & front matter gotchas

- Front matter is the block between two `---` lines at the top of a file. It is **not**
  rendered. Editing front matter changes the page's metadata/permalink/layout — not its
  visible content.
- `{{ page.X }}` reads from the current page's front matter; `{{ site.X }}` reads from
  `_config.yml`.
- `{{ content }}` in a layout is where the content file's body is injected. Don't
  duplicate content into the layout.
- `{% include x.html %}` brings in a partial. Editing the include affects every page
  that uses it.
- `_layouts/job.html` uses `{% case page.url %}` — without an arm matching the job's
  permalink, the body include never renders. Always add an arm when adding a job.
- `_includes/navigation.html` currently has its `<ul>` of nav links HTML-commented out;
  `_config.yml`'s `links:` array is therefore unused at runtime. If asked to add/edit a
  nav link, uncomment the block first and confirm with a build.
- Footer social icons in `_includes/footer.html` are each gated on
  `{% if site.owner.<network> %}`. To hide an icon, clear the value in `_config.yml`
  rather than deleting the markup. To add one, set the value.
- The post layout `_layouts/post.html` line 80 has `{% if page.share contains
  'invalid-made-up-by-jaisen' %}` — that gate is intentionally false to suppress
  share buttons. Don't change it unless that's the explicit instruction.

## 9. Verification

Before committing:

1. Build: `bundle exec jekyll build`. This writes to `_site/`. A non-zero exit means
   you broke Liquid or front matter.
2. Confirm your edited file is the one in the load path:
   - Posts/pages: find the rendered HTML in `_site/<permalink>/index.html` and confirm
     it contains your change.
   - Job pages: `_site/work-at-<name>.html`.
   - Homepage: `_site/index.html`.
3. If you edited CSS, confirm the URL referenced in the rendered `<head>` matches the
   file you edited (`/assets/less/main.css` for Jekyll-rendered pages,
   `/assets/home/css/style.css` for the homepage).
4. Do **not** commit `_site/` or `.jekyll-cache/`.

You do not need to run `grunt`. The Grunt tasks (`recess`, `uglify`, `imagemin`) are
dormant in this repo and edit-to-asset wiring already bypasses them.

---

## 10. Hard rules

- Never edit `_site/`, `.jekyll-cache/`, `assets/less/*.less`, `assets/css/main.css`,
  `assets/js/scripts.min.js`, or `lattice.js`.
- Never delete a Liquid tag or `{% include %}` to make a layout match desired output —
  fix the content file or the include instead.
- Never alter front matter keys (`layout`, `permalink`, `title`) unless the instruction
  is explicitly about routing/metadata.
- Never change `_config.yml`'s `permalink:` pattern.
- Never add new dependencies (Gemfile, package.json) for a content/design edit.
- Never add a new top-level page without front matter — Jekyll won't process it.
- One edit per request. If the instruction can't be satisfied by editing one source
  region, prefer the smallest set of files and explain the spread in your commit
  message.
